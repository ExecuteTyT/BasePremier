## Root Causes

### 1. Three.js bundle (~2.3MB uncompressed / ~400KB gzip)

The homepage loads the 3D MonogramBP component which pulls in three.js + @react-three/fiber.

- `three_build_three_core`: 1518KB
- `three_build_three_module`: 842KB
- `@react-three/fiber`: 817KB

Even with `dynamic({ ssr: false })`, these chunks load eagerly because MonogramBP is in the above-fold hero section.

**Mitigation:** Delay 3D scene load with `IntersectionObserver` or `setTimeout(500)` after hero paints. On mobile, skip entirely (already `hidden md:block`).

### 2. next-devtools (730KB) in production bundle

`next/dist/compiled/next-devtools` is bundled by Turbopack even in production. `devIndicators: false` does not prevent this. This appears to be a Next.js 16.2.4 Turbopack-specific issue.

**Mitigation:** Try webpack build (`next build --no-turbo` or `turbopack: false` in config when stable). Or wait for Next.js fix.

### 3. Unminified JS (~557KB savings per Lighthouse)

Lighthouse detects JS chunks without optimal minification. Possible Turbopack production quirk.

### 4. LCP element delay

LCP at 13.9s vs FCP at 2.1s indicates the largest painted element appears very late. Likely the hero H1 via CharReveal (GSAP animates it from yPercent:110 — visible but off-screen via overflow-hidden — but GSAP itself needs to load first from the 687KB motion-dom chunk).

**Fix applied:** Removed `opacity: 0` from GSAP fromTo. Chars are always opacity:1, only clipped by overflow-hidden parent until GSAP animates yPercent.

## Next Steps (Performance)

1. **Replace Three.js lazy-load strategy**: delay render of MonogramBP by 500ms after initial paint using `useEffect + setTimeout`. This lets LCP (H1) complete before Three.js parsing begins.
2. **Build with webpack**: test `turbopack: false` in next.config.ts to see if next-devtools is removed and minification improves.
3. **Add Brotli/gzip on Nginx**: will halve the effective transfer size on real VPS.
4. **Real-world benchmark**: these numbers are simulated Slow 4G. Run a real audit on the deployed VPS to get accurate metrics.
