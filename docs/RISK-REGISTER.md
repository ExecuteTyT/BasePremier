# RISK-REGISTER.md — Реестр рисков BASE Premier

> Систематический список рисков проекта, упорядоченный по приоритету. Каждый риск имеет: вероятность × ущерб = приоритет, владельца, план митигации, точку триггера.
>
> **Особенность проекта:** solo-founder workflow через Claude Code. Это создаёт специфические риски, которых нет в командных проектах — они вынесены в отдельную секцию §6.

## Легенда

| Приоритет | Вероятность × Ущерб |
|---|---|
| 🔴 Critical | Высокая × Высокий |
| 🟠 High | Высокая × Средний / Средняя × Высокий |
| 🟡 Medium | Средняя × Средний |
| 🟢 Low | Низкая × Любой / Любая × Низкий |

## 1. Контентные риски

### R-001 🔴 Отсутствие профессиональных фото 9 из 10 мастеров

- **Вероятность:** 100% (только 1–2 кадра существует)
- **Ущерб:** Высокий — карточки мастеров будут с placeholder, страница `/barbers` теряет 70% impact
- **Триггер:** Фаза 4, нед. 8 — нужны реальные фото для интеграции в Sanity
- **Митигация:**
  1. **Основной план:** организовать фотосессию в Казани к нед. 7 (бюджет 30–80 тыс ₽, 1 смена). Поиск фотографа: Behance + Telegram-чаты Казань
  2. **План B:** сначала запускаемся с ч/б монохромными силуэтами через Midjourney (тон сохраняется, лица анонимные), фото добавляем по мере готовности
  3. **План C:** буква-инициал в стиле логотипа BP как аватар (Аesop-style абстракция)
- **Триггер для эскалации:** если к нед. 6 нет утверждённого фотографа → переходим на План B и информируем заказчика
- **Владелец:** заказчик (организация съёмки) + Claude Code (продакшн альтернатив)

### R-002 🟠 Тексты не «продают» — нейтральный AI-тон

- **Вероятность:** Средняя — Claude может выдавать «корректные, но безликие» тексты
- **Ущерб:** Высокий — главный страх заказчика «красиво, но не продаст» материализуется
- **Триггер:** Фаза 5, нед. 13 — после генерации первых текстов
- **Митигация:**
  1. Жёсткая привязка ко всем 6 anti-positioning-правилам в каждом промпте к Claude
  2. A/B-тестирование 3 hero-вариантов через Я.Метрику после запуска (4 недели данных)
  3. Перед публикацией — обязательная проверка по чек-листу из CONTENT-BRIEF (тон, длина, эпитеты)
  4. Использовать `agent: copywriter` с system-промптом, заточенным под Tom Ford-style сдержанность
  5. Заказчик сам читает все ключевые тексты вслух — если звучит как chatGPT-ответ → переписываем
- **Владелец:** Claude Code + финальная вычитка заказчиком

### R-003 🟡 Биографии мастеров не получены вовремя

- **Вероятность:** Средняя — собирать анкеты с 10 человек требует усилий
- **Ущерб:** Средний — карточки мастеров публикуются с шаблонными био
- **Триггер:** Нед. 5 — био должны быть собраны
- **Митигация:**
  1. Заказчик создаёт Google-форму с 5 вопросами и рассылает мастерам через Telegram-чат заведения
  2. Дедлайн на сбор — 7 дней
  3. План B: Claude генерирует «безликие» профессиональные био на основе должности и счётчика отзывов, обновляем по мере получения реальных
- **Владелец:** заказчик (сбор) + Claude Code (форматирование)

### R-004 🟡 Отсутствие согласий клиентов на отзывы

- **Вероятность:** Средняя — клиенты могут не отвечать
- **Ущерб:** Средний — теряем трогательную «лицевую» проф-копию, остаётся анонимная
- **Триггер:** Нед. 13 — отзывы в Sanity
- **Митигация:** запасной план — все отзывы в формате «А., 32 года» с ссылкой на Я.Карты для верификации
- **Владелец:** заказчик

## 2. Технические риски

### R-005 🔴 Баланс анимаций vs Performance/LCP

- **Вероятность:** Высокая — заказчик выбрал «МАКСИМУМ анимаций» + Awwwards-уровень
- **Ущерб:** Высокий — LCP > 2.5s срывает SEO в Я и ухудшает CR
- **Триггер:** Нед. 7, 11, 14 — Lighthouse-аудиты на главной
- **Митигация:**
  1. **Performance-бюджет** в `next.config.ts` enforced: bundle ≤ 200 KB gzip, hero LCP ≤ 250 KB
  2. **3D BP** — lazy через `dynamic({ ssr: false })`, на mobile — статичный PNG
  3. **Hero видео** — preload только poster, видео грузится по `requestIdleCallback`
  4. **GSAP-плагины** — импортировать только нужные (`ScrollTrigger`, `SplitText`), не весь GSAP
  5. **Lenis** — `smoothTouch: false` (на mobile нативный скролл)
  6. **Анимации с гардами** `prefers-reduced-motion` для всего нелинейного
  7. **Lighthouse CI** — gate на каждый PR, build не проходит при Performance < 85
  8. **Real User Monitoring** — Plausible (self-hosted на VPS) или Yandex.Metrica RUM
- **Эскалация:** если в нед. 14 LCP > 3.5s → отключаем 3D BP в hero (статика), перепрофилируем под Lighthouse 90+
- **Владелец:** Claude Code (perf-auditor agent)

### R-006 🟠 Кастомизация YClients widget недоступна на текущем тарифе

- **Вероятность:** Средняя — кастомизация требует тарифа Plus
- **Ущерб:** Средний — виджет в стандартной теме YClients, разрушает «премиум»-впечатление
- **Триггер:** Нед. 12 — интеграция виджета
- **Митигация:**
  1. **До нед. 6** уточнить у YClients API: что доступно на текущем тарифе и сколько стоит upgrade на Plus (ориентировочно +5–10 тыс ₽/мес)
  2. **План B:** обернуть iframe в нашу модалку с правильным фоном (`--bg-primary`), что частично скрывает чужой стиль
  3. **План C:** отказаться от виджета YClients и реализовать свою форму, которая через API создаёт запись (требует больше работы, но лучше контроль UX)
- **Владелец:** Claude Code + согласование тарифа с заказчиком

### R-007 🟠 YClients API rate limits / downtime

- **Вероятность:** Средняя — публичное API не гарантирует SLA
- **Ущерб:** Средний — live occupancy перестаёт работать, сайт показывает старые данные или ошибку
- **Триггер:** В любой момент после запуска
- **Митигация:**
  1. **Кеш на Edge Runtime** — TTL 60 сек, при ошибке отдаём stale-while-revalidate
  2. **Sentry alerts** — если API даёт 5xx > 5 раз за 5 минут, нотификация в Telegram
  3. **Graceful fallback** — при недоступности API в UI показываем «загруженность уточняйте по телефону»
- **Владелец:** Claude Code

### R-008 🟡 Sanity webhook revalidation сбивается → старый контент

- **Вероятность:** Низкая — Sanity webhook надёжен
- **Ущерб:** Средний — заказчик публикует новую статью, она не появляется на сайте
- **Триггер:** После любой публикации в Sanity Studio
- **Митигация:**
  1. **Backup путь:** ISR с 1-часовым revalidate — даже при сломанном webhook контент обновится через час
  2. **Manual revalidate** — кнопка в админке заказчика на специальной странице `/admin/revalidate?secret=...`
  3. **Monitoring webhook calls** через Sanity logs
- **Владелец:** Claude Code

### R-009 🟡 Sound design мешает UX → пользователи раздражаются

- **Вероятность:** Средняя — звук в web обычно неуместен
- **Ущерб:** Средний — bounce rate растёт
- **Триггер:** После запуска, через 1–2 недели данных
- **Митигация:**
  1. **Sound по умолчанию ВЫКЛ** — флаг включается явно через тумблер
  2. **Тумблер всегда виден** в правом нижнем углу
  3. **Звуки тихие** — peak ≤ -18 dB, никаких джинглов, только subtle interactions
  4. **Метрика** — события `sound_on`/`sound_off` для понимания, кто включает
- **Владелец:** Claude Code

### R-010 🟡 Кастомный курсор ломается на специфичных устройствах

- **Вероятность:** Средняя — touch-устройства, гибридные ноуты, accessibility-инструменты
- **Ущерб:** Средний — UX сломан, нативный курсор скрыт, пользователь не видит указатель
- **Триггер:** После запуска
- **Митигация:**
  1. **Включается только при `(pointer: fine) and (any-hover: hover)`** через media query
  2. **Тестирование** на iPad с трекпадом, ноутбуке с touchscreen
  3. **Глобальный kill-switch** — если у пользователя `prefers-reduced-motion` или баг — `cursor: auto !important` через CSS-override
- **Владелец:** Claude Code

## 3. SEO и миграционные риски

### R-011 🟠 Падение позиций при миграции со старого сайта

- **Вероятность:** Средняя (хотя позиции и так на 8–9 странице — терять особо нечего)
- **Ущерб:** Средний — задержка раскрутки нового сайта на 2–3 месяца
- **Триггер:** Через 1–2 недели после смены DNS
- **Митигация:**
  1. **Аудит старых URL** перед миграцией (через Я.Вебмастер + Screaming Frog), составление полного списка
  2. **301-редиректы** прописаны в `next.config.ts` для всех старых URL
  3. **Я.Вебмастер** — ручная отправка нового sitemap.xml + переобход страниц
  4. **Сохранение URL-структуры** там, где это возможно (`/services` остаётся `/services`)
  5. **Мониторинг** через Я.Вебмастер 2 месяца после запуска
- **Эскалация:** если позиции упали > 20 мест → SEO-аудит и точечные правки
- **Владелец:** Claude Code (seo-auditor agent)

### R-012 🟡 Новые страницы не индексируются в Я

- **Вероятность:** Средняя
- **Ущерб:** Средний — потеря органики со статей блога
- **Триггер:** Через 4 недели после публикации
- **Митигация:**
  1. **sitemap.xml автогенерация** + ручная отправка в Я.Вебмастер
  2. **Я.Вебмастер «переобход страниц»** для каждой новой статьи
  3. **Internal linking** — каждая новая статья получает ссылки минимум с 2 других страниц
  4. **JSON-LD Article + BreadcrumbList**
  5. **OG meta + Twitter cards** для расшаривания
- **Владелец:** Claude Code

## 4. Юридические и комплаенс-риски

### R-013 🟡 Cookie-banner не соответствует требованиям

- **Вероятность:** Низкая — банальный banner проходит большинство аудитов
- **Ущерб:** Низкий — потенциальный штраф РКН (редкость)
- **Митигация:**
  1. Banner с двумя кнопками: «Принять все» и «Только необходимые»
  2. Текст согласия включает упоминание Я.Метрики и других трекеров
  3. Ссылка на политику конфиденциальности на `/legal/privacy`
- **Владелец:** Claude Code + (опционально) юрист

### R-014 🟡 Защита персональных данных в формах обратной связи

- **Вероятность:** Низкая (заказчик решил не собирать ПД)
- **Ущерб:** Высокий при нарушении (152-ФЗ, штрафы РКН до 100 тыс ₽)
- **Триггер:** Любая форма на сайте, собирающая телефон/email/имя
- **Митигация:**
  1. **На старте:** только webhooks в Telegram, без сохранения ПД на нашем сервере
  2. **Если форма обратной связи будет:** обязательный чекбокс согласия, отдельная политика, без хранения в базе (только проксирование в Telegram-бот)
  3. **При накоплении ПД:** регистрация ИП как оператора в РКН (заполнить форму на rkn.gov.ru, бесплатно, 1 час)
- **Владелец:** заказчик + Claude Code

### R-015 🟢 Шрифты — лицензионная чистота

- **Вероятность:** Очень низкая — выбраны бесплатные шрифты Google Fonts
- **Ущерб:** Нулевой — все три шрифта под SIL Open Font License (OFL)
- **Триггер:** Использование шрифта без проверки лицензии
- **Митигация:**
  1. Используем **только** Fraunces + Inter + JetBrains Mono — все три из Google Fonts, OFL-1.1
  2. Подключение исключительно через `next/font/google` — гарантирует self-hosting CDN-шрифтов и легальное использование
  3. Никаких mirror-репозиториев и no-name шрифтов с бесплатных сайтов (DaFont/FontSpace) без проверки лицензии
  4. Если в будущем потребуется платный display-шрифт (Tier 1: PP Neue Montreal/Editorial New) — покупка лицензии и хранение в `docs/licenses/` (PDF EULA)
- **Владелец:** Claude Code (контроль), пользователь (финал)

## 5. Бизнес-риски

### R-016 🟠 Сайт не выстреливает, конверсия не растёт

- **Вероятность:** Средняя — Awwwards-сайты не всегда конверсионны
- **Ущерб:** Высокий — заказчик потерял время и деньги
- **Триггер:** 2 месяца после запуска, если CR не вырос
- **Митигация:**
  1. **A/B-тесты** через Я.Метрику с первой недели (3 варианта hero copy)
  2. **Heatmap-анализ** — где люди реально кликают, что игнорируют
  3. **Корректировка под данные** — упрощаем секции, которые не дают результата
  4. **План отката** — старый сайт остаётся доступен, в худшем случае возвращаемся
- **Владелец:** заказчик (анализ метрики) + Claude Code (правки)

### R-017 🟡 Я.Директ — высокий CPL при отсутствии истории кампаний

- **Вероятность:** Высокая — заказчик никогда не запускал Директ
- **Ущерб:** Средний — первые 4–8 недель CPL высокий (1500–3000 ₽), потом стабилизация
- **Триггер:** Нед. 16 — запуск Директа
- **Митигация:**
  1. **Стартовый бюджет** — 30 тыс ₽/мес (нижняя граница плана), не больше до получения данных
  2. **Узкий таргет** на старте: только Казань + конкретные ключи «барбершоп казань», «мужская стрижка казань премиум»
  3. **Минус-слова** проработать заранее (детская стрижка, дешёвый, бесплатно, акция)
  4. **Цели в Метрике** настроены — Директ оптимизируется по реальным конверсиям
  5. **Через 6 недель** — анализ, если CPL > 2000 ₽ → пересборка кампаний
- **Владелец:** Claude Code (настройка) + заказчик (контроль бюджета)

### R-018 🟢 Заказчик хочет «пере-впихнуть» функциональность после запуска

- **Вероятность:** Высокая — нормально для собственника продукта
- **Ущерб:** Низкий — если правильно фильтровать через roadmap
- **Митигация:**
  1. **После-запуска backlog** в репо (`docs/BACKLOG.md`)
  2. **Каждая новая идея** проходит через критерий «закрывает ли реальную проблему по данным метрики»
  3. **Релизы раз в 2 недели**, минор-фичи — пакетами
- **Владелец:** заказчик + Claude Code

## 6. Solo-Founder риски (специфика этого проекта)

> Эти риски возникают именно из-за того, что весь проект делает один человек с Claude Code. В команде они компенсируются ролями.

### R-101 🔴 Single point of failure — заказчик заболевает / выгорает

- **Вероятность:** Средняя — 16 недель проект, пиковые нагрузки на нед. 5–7 и нед. 13–15
- **Ущерб:** Высокий — проект встаёт, нет коллег, которые подхватят
- **Митигация:**
  1. **Все артефакты в репо** — другой человек может подхватить, имея только этот пакет docs/
  2. **CLAUDE.md и ROADMAP.md** написаны так, что любой Claude Code может продолжить с любого тикета
  3. **Резерв сроков** — 16 недель уже включают 4 недели запас «на качество», есть возможность сдвинуть
  4. **Регулярные «save points»** — после каждой фазы коммит + push + tag, чтобы можно было откатиться
- **Владелец:** заказчик (личное здоровье) + проектный архив

### R-102 🟠 Технический долг растёт — нет код-ревью

- **Вероятность:** Высокая — solo-режим без второй пары глаз
- **Ущерб:** Средний — код становится сложно поддерживать через 6 месяцев
- **Митигация:**
  1. **CLAUDE.md правила** — строгие (TypeScript strict, no any, conventional commits, бюджеты)
  2. **Hooks автоматизации** — pre-commit `lint + typecheck + format`, не дают коммитить мусор
  3. **Subagents для аудитов** — каждые 2 недели запуск `code-quality-auditor` через worktree
  4. **Стандартизация через skills** — все секции делаются через одну и ту же процедуру
  5. **Минимальные тесты** — Playwright E2E на критичные флоу (запись)
- **Владелец:** Claude Code (дисциплина) + квартальные ревью

### R-103 🟠 Риск «увидел дерево, потерял лес» — слишком глубоко в коде

- **Вероятность:** Высокая — solo-разработчик легко уходит в полировку деталей
- **Ущерб:** Средний — фичи откладываются, дедлайны сдвигаются, бюджет тратится на «не важное»
- **Митигация:**
  1. **ROADMAP.md как источник истины** — каждый день начинается с проверки «что в плане сегодня»
  2. **Прогресс-трекер** — milestone после каждой фазы, blocking-thresholds
  3. **Time-boxing** — каждый тикет имеет оценку, превышение в 2× → пересмотр подхода
  4. **«Готово — лучше идеально»** — каждый тикет должен быть закрыт, потом улучшается итерациями
- **Владелец:** заказчик (дисциплина) + checkpoints каждую пятницу

### R-104 🟡 Зависимость от одного провайдера — Anthropic API падает

- **Вероятность:** Низкая — Anthropic Claude стабилен, но не 100%
- **Ущерб:** Средний — работа встаёт на час-сутки
- **Митигация:**
  1. **Status page** Anthropic — мониторим
  2. **Локальные fallback модели** на случай долгого даунтайма (Cursor + GPT-4 / Gemini как резерв)
  3. **Не в критичный момент** — не делаем deploy перед известными maintenance windows
- **Владелец:** заказчик

### R-105 🟡 Чрезмерное усложнение — Claude любит «over-engineering»

- **Вероятность:** Средняя — AI часто добавляет паттерны, которых не нужно
- **Ущерб:** Средний — 200 строк там, где можно 30; сложно дебажить через 3 месяца
- **Митигация:**
  1. **CLAUDE.md правило** — «KISS, никаких абстракций до 3-го повторения»
  2. **Каждый PR** — review своими глазами с вопросом «нужно ли это?»
  3. **Метрика — bundle size** — если новая фича добавила +20 KB → пересмотр
  4. **Ограничение на dependency** — никаких новых пакетов без обсуждения
- **Владелец:** Claude Code + заказчик

### R-106 🟢 Поддержка после запуска — кто будет дописывать через год

- **Вероятность:** Высокая — все проекты требуют поддержки

## 9. Регулярная переоценка рисков

- **Каждую пятницу** — 30 минут на пересмотр risk-register
- **После каждой фазы** — все «открытые» риски проверяются: остаются / снимаются / появляются новые
- **При материализации риска** — фиксируем в `docs/incidents/` с разбором

## 10. Open communication channel

Все риски, которые материализуются → обсуждаются с заказчиком в день обнаружения, не накапливаются. Принцип «no surprises by Friday».
